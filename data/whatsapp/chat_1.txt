12/08/2025, 00:39 - Abrar...: 4.x USE CASE DIAGRAM
The Use Case Diagram of TruthScope represents the interaction between the system and its external actors, describing the core functionalities offered to users and the flow of data through the system. It provides a high-level overview of what the system does, without going into internal implementation details.

Actors
User (Primary Actor) – The individual who enters a claim for verification, browses news updates, and views fact-checking results.

Chrome Extension – Acts as an alternative client for users to fact-check directly within their web browser.

External APIs – Supporting systems such as Google Custom Search API and NewsAPI that supply evidence for claim verification.

System Scheduler – An implicit actor that triggers cache refreshes and scheduled news updates.

Use Cases
Submit Claim for Verification
The user enters a claim via the web interface or Chrome Extension. The system preprocesses the text (cleaning, NER, keyword extraction, intent and region identification), checks the cache for prior results, and if not found, retrieves relevant evidence from external APIs. This evidence is analyzed semantically, scored, and a verdict is generated.

View Verification Results
The system presents truth and confidence scores, a color-coded verdict, and a breakdown of supporting and contradicting sources with direct links.

Browse News Dashboard
The user accesses categorized, up-to-date news with AI-generated summaries and relevant images.

Verify Text via Chrome Extension
The extension sends selected webpage text to the backend for fact-checking, returning inline verdicts without leaving the current page.

Inspect Source Breakdown
The user can view details of each source’s credibility, stance, and publication time, ensuring transparency.

Refresh News Feed
Triggered either manually or automatically by the scheduler to ensure the news content is current.

Importance in the Project
The Use Case Diagram ensures that all functional requirements are clearly defined and that every actor-system interaction is mapped. This allows developers to verify completeness of the system’s capabilities before moving into implementation. For TruthScope, this diagram also highlights the integration points with external services, the Chrome Extension, and the user-facing components.

4.x CLASS DIAGRAM
The Class Diagram of TruthScope depicts the structural design of the system, detailing its classes, attributes, methods, and relationships. It represents the static architecture, ensuring that every component in the system has a defined role and communicates with the correct modules.

Core Classes
Claim – Represents a user’s submitted statement. Attributes include rawText, cleanText, language, intent, and region. Methods perform preprocessing tasks such as entity extraction and keyword generation.

Evidence – Stores retrieved articles or content used to verify claims, including url, title, sourceName, and content.

SourceMeta – Maintains metadata for each source, including credibility, expertise, region relevance, and reach.

AnalysisResult – Encapsulates the computed truth score, confidence score, verdict, and summary for display to the user.

SourceAssessment – Represents the stance (support, contradict, neutral) of each piece of evidence towards the claim.

Service Classes
Preprocessor – Handles claim text cleaning, language detection, NER, and intent identification.

EvidenceRetriever – Interfaces with Google CSE, NewsAPI, and the Scraper to gather relevant evidence.

SemanticAnalyzer – Uses transformer models to assess semantic similarity and determine stance.

ScoringEngine – Computes truth and confidence scores using weighted source metadata.

CacheStore – Stores results temporarily to reduce redundant API calls.

VerifierFacade – The orchestrator class that connects preprocessing, retrieval, semantic analysis, and scoring to produce the final result.

Integration Clients
GoogleCSEClient and NewsAPIClient – Specialized connectors for external APIs.

Scraper – Extracts clean text from web pages.

Presentation Layer
ResultsFormatter – Formats analysis results for frontend consumption.

FrontendAdapter – Prepares UI components for the React web app.

ChromeExtensionAdapter – Sends formatted fact-checking results back to the Chrome Extension interface.

Key Relationships
VerifierFacade uses Preprocessor, EvidenceRetriever, SemanticAnalyzer, ScoringEngine, and CacheStore.

EvidenceRetriever depends on GoogleCSEClient, NewsAPIClient, and Scraper.

SemanticAnalyzer uses EmbeddingModel and LLMReasoner for NLP tasks.

AnalysisResult is formatted by the presentation layer before being shown to the user.

Importance in the Project
The Class Diagram ensures a modular, maintainable system design. By defining clear responsibilities for each class, it supports scalability, simplifies debugging, and enables future enhancements such as adding multi-language support or integrating image/video fact-checking. For TruthScope, the diagram also reinforces the separation between backend processing, data retrieval, and frontend presentation, ensuring a clean architecture.

- Reduce redundant processing.

👉 **Goal**:
- More accurate semantic matching.

👉 **Goal**:
- Truth score: derived from support vs contradiction articles.
- Confidence score: weighted heavily by source credibility, not count.

🛠️ Hint:
- Redesign `confidence_scorer.py` to:

A Knowledge Graph is a structured way of representing information using nodes and edges. Nodes represent real-world entities — like people, locations, and tasks — and edges describe the relationships between them, such as “located in,” “task due,” or “works at.” What makes a knowledge graph powerful is that it doesn’t store facts in isolation. Instead, it captures the connections between concepts, allowing systems to understand context, reason across related information,

The first paper, Knowledge Graph Inference for Spoken Dialog Systems is a conference paper from April 2015 which applies this structured knowledge to spoken dialog systems. Traditional SDS treat state or context tracking and database lookup as two separate steps which are often brittle, especially with speech recognition errors. IKG solves this by converting an existing semantic knowledge graph into a Markov Random Field, where each node becomes a probabilistic variable representing how likely it is relevant to the user’s goal. When the user mentions attributes these attributes become evidence nodes that influence connected entities. Inference then spreads this likelihood through the graph, ranking entities or nodes on probability. This unifies state tracking and retrieval in one step. Experiments using Microsoft’s Satori graph showed that IKG outperforms traditional lookup systems — especially when input is noisy — because the graph structure naturally supports more robust reasoning.

The second paper, DiffKG — Dialogue Differentiable Knowledge Graph, takes KG reasoning a step further by integrating it directly into a transformer model. Instead of converting the graph into text, DiffKG allows the model to walk the graph in a differentiable way. The architecture includes a dialogue encoder that interprets the conversation, a differentiable KG reasoning module that performs learned multi-hop traversal, a logical operations module for filtering and constraints, and a response decoder that produces both the system action and the natural language response. The key contribution is that the reasoning steps are fully interpretable — the model can show which nodes and relations it followed to arrive at an answer. DiffKG achieves strong performance on datasets like SMD and OpenDialKG while using less computation than text-flattening approaches.
Together, IKG and DiffKG show two complementary ways to integrate knowledge graphs into dialogue systems — one through probabilistic inference and the other through differentiable graph traversal — both moving toward more intelligent, transparent, and context-aware conversational AI.
